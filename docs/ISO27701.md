# ISO/IEC 27701 compliance roadmap for Litmo

**Status:** Planning / engineering alignment (not certification)  
**Date:** 2026-07-13  
**Audience:** Founder, engineering agents, future privacy/security reviewers  
**Not legal advice and not a claim of ISO 27701 certification.**

This roadmap explains what ISO/IEC 27701 expects, how Litmo already maps, and a  
**small-team, phased plan** to close gaps—with emphasis on:

1. Privacy by design  
2. Consent management  
3. Data subject rights  
4. Integration with existing **safety** features (Soft Signal, reports, RLS, dual snapshots)

Companion docs: `docs/GDPR.md`, `docs/DATA_CLASSIFICATION.md`,  
`docs/PRIVACY_POLICY_DRAFT.md`, `docs/KNOWN_LIMITATIONS.md`, ADR 0011 / 0042 / 0050–0052.

## 1. What ISO/IEC 27701 is

**ISO/IEC 27701** specifies requirements and guidance for a **Privacy Information  
Management System (PIMS)**. In practice it:

- **Extends ISO/IEC 27001** (information security management) and  
  **ISO/IEC 27002** (security controls) with **privacy-specific** requirements.  
- Is typically **certified as an extension of an ISO 27001 ISMS**, not as a  
  stand-alone certificate in most schemes.  
- Distinguishes **PII controllers** (determine purposes/means of processing) from  
  **PII processors** (process PII on behalf of a controller). Litmo’s operator  
  is primarily a **controller** of user account, preference, and session data;  
  vendors (e.g. Supabase, Apple) act as **processors / subprocessors** for  
  infrastructure services.

**Relation to GDPR:** ISO 27701 is a **management-system** standard. It does  
**not** replace GDPR. Done well, a PIMS produces evidence that supports  
accountability (Art. 5(2), Art. 24–32 GDPR): policies, records, risk treatment,  
and operational controls. Litmo’s GDPR engineering work (`docs/GDPR.md`) is a  
**parallel legal-alignment track**; this document is the **management-system  
roadmap**.

**2025 edition note:** A newer ISO/IEC 27701 edition is circulating in industry  
guidance with reorganized controller control sets (often summarized as ~31  
controller-focused controls covering conditions for processing, obligations to  
principals, and privacy by design). This roadmap uses **stable control themes**  
that remain valid under both 2019-era Annex A/B thinking and 2025-style  
groupings. When you hire a certifying body, lock the **edition year** they will  
audit against.

## 2. Scope recommendation (small team)

### 2.1 In scope for Litmo PIMS (initial)

| In scope | Rationale |
| -------- | --------- |
| Mobile app (Expo/iOS) + Supabase backend | Primary personal-data systems |
| Authenticated adult users | Core data subjects |
| Touch Language, consent preferences, Consent Snapshots, sessions | Highest sensitivity / product core |
| Safety ops (reports, blocks, restrictions, staff review) | High risk if mishandled |
| Device-local stores (quizzes, learning, E2E keys, ND prefs) | PII / preferences on endpoints |
| Passkey / Keychain / SensitiveAccessGate | Security controls protecting PII |

### 2.2 Explicitly out of scope initially

| Out of scope | Why |
| ------------ | --- |
| Public App Store production multi-jurisdiction launch | Requires legal + full retention/erasure |
| macOS Ops staff org-wide HR systems | Separate ISMS scope if ever certified |
| Third-party clinical / emergency integrations | Not product |
| Full ISO 27001 enterprise SOCs | Too heavy; use **lightweight ISMS subset** first |

### 2.3 Controller vs processor statement (draft for SoA)

- **Litmo operator:** PII **controller** for account, profile, touch/consent  
  versions, session snapshots, safety cases it opens.  
- **Supabase / hosting / auth DB:** PII **processor** under contract (DPA  
  required before certification).  
- **Apple:** Passkeys/biometrics—platform processor for credential material;  
  Litmo does not store biometric templates.  
- **Users’ partners (session counterpart):** Not processors; limited shared  
  access to **snapshot semantics** only by product design.

## 3. Core requirements (practical digest)

ISO 27701 expects you to **establish, implement, maintain, and continually  
improve** a PIMS. For a small team, translate that into four workstreams:

### A. Governance (PIMS clauses / ISMS integration)

| Expectation | Small-team practice |
| ----------- | ------------------- |
| Defined scope | This file §2 |
| Roles & responsibilities | Founder = privacy owner; engineering = control implementers; named legal/privacy reviewer before external alpha |
| Privacy risk assessment | Lightweight DPIA-style register for touch/consent/snapshots/safety (§6) |
| Statement of Applicability (SoA) | Spreadsheet of controls: Apply / Partial / N/A + evidence path |
| Documented policies | Repository ADRs + GDPR.md + this roadmap; not a separate wiki farm |
| Continual improvement | Changelog + KNOWN_LIMITATIONS + incident notes |
| Competence | Short privacy checklist for contributors in AGENTS.md / PR template |

### B. Conditions for processing (lawful basis, purpose, consent, records)

| Expectation | Litmo today | Gap for 27701 evidence |
| ----------- | ----------- | ---------------------- |
| Purpose limitation | Product constitution; no ads SDKs | Written purpose inventory per processing activity |
| Lawful basis documented | Intent in GDPR.md | Legal confirmation + RoPA (record of processing) |
| Consent where used | Session dual-confirm, privacy notice ack, quiz share/compare | Consent records with version, time, purpose, withdraw path |
| Contracts with processors | Informal | DPAs with Supabase/Apple/etc. |
| Joint controllers | None known | N/A unless partnerships |
| Records of processing | Implicit in schema + DATA_CLASSIFICATION | One RoPA table (even a markdown table) |

### C. Obligations to PII principals (transparency + rights)

| Right / duty | Litmo today | Gap |
| ------------ | ----------- | --- |
| Transparent notice | In-app Privacy Policy + Data Protection | Named controller, contacts, retention windows |
| Access / portability | `export_my_data` + device inventory | Completeness attestation; SLA |
| Rectification | Profile / touch / consent edit | Documented request channel |
| Erasure | Device wipe + `request_account_erasure` queue | Fulfillment runbook; hard-delete still blocked |
| Restriction / objection | Soft Signal, blocks, holds | Explicit “restrict processing” edge cases |
| Withdraw consent | Session withdraw; quiz withdraw | Notice of consequences |
| Response time | Not formalized | 30-day (or local law) tracker |
| Identity verification | Auth session | Document for out-of-band requests |

### D. Privacy by design and by default

| Expectation | Litmo today | Gap |
| ----------- | ----------- | --- |
| Collection limitation | Discovery field gate; demo synthetic | Formal collection checklist in PR reviews |
| Minimization | RLS, no discovery of private notes | Periodic field inventory |
| Accuracy | Versioned profiles | User-facing correction help |
| Retention limits | POC cascades; purge helpers | Production schedules + jobs owned |
| De-identification | Limited (trust facts, not scores) | Define when anonymization applies |
| Temporary files / caches | Expo/local | Document crash/log policy |
| Secure transmission | TLS via Supabase | Certificate/pinning review later |
| Encryption | ADR 0011 device-bound notes; E2E quiz packages | Key lifecycle SOP |

## 4. Mapping: focus areas × Litmo product

### 4.1 Privacy by design (priority)

| Control theme | Existing Litmo assets | Next practical actions |
| ------------- | --------------------- | ---------------------- |
| Default-deny discovery | Discovery function whitelist | ADR note when adding any new public field |
| Sensitive free text | App-encrypted notes, Face ID gates | Keep fail-closed; never log plaintext |
| Snapshots | Server-authoritative, dual confirm, no private notes inside | Document snapshot retention after session end |
| Partner quiz E2E | Ciphertext-only relay refuse-list | No plaintext weather on server |
| Logging | `privacySafeMetadata` | Add CI grep for forbidden log keys |
| Demo isolation | No server account in demo | Banner: synthetic data only |

**Small-team default:** Every PR that touches schema or export must update  
`DATA_CLASSIFICATION.md` and, if sensitive, `GDPR.md` / this SoA row.

### 4.2 Consent management (product consent ≠ only GDPR consent)

Litmo has **two layers** of “consent”—ISO 27701 cares about **both**, but they  
must not be conflated in UX or audits:

| Layer | Meaning | Product implementation | PIMS evidence |
| ----- | ------- | ---------------------- | ------------- |
| **A. Privacy / processing consent & bases** | Legal permission to process PII | Privacy notice version + `accept_privacy_notice`; bases draft in GDPR.md | Notice version log; acceptance table |
| **B. Session / touch consent** | Product safety: mutual, current, revocable permission to touch | Touch Language versions, dual Consent Snapshot, Soft Signal, withdraw | Snapshot rows, lifecycle events, ADRs |
| **C. Optional feature consent** | Partner quiz share/compare; optional quiz backup | Four-gate share/compare; owner-only quiz backup | Invite/compare state; export categories |

**Integration rule for safety features:**

- Soft Signal and session withdrawal are **safety rights** and also support  
  **withdrawal of processing for that session context**.  
- Matching holds / bans are **safety restrictions**, not a substitute for  
  erasure. Record purpose (safety) separately from erasure requests.  
- Reports to moderators: minimize content; private notes not re-exported to  
  peers; staff access audited (staff auth still incomplete—see limitations).

**Practical consent register (minimum viable):**

| Purpose ID | Processing | Mechanism | Withdraw |
| ---------- | ---------- | --------- | -------- |
| `acct` | Account & auth | Contract / account creation | Erasure request + sign-out |
| `touch_consent_pref` | Store touch/consent versions | Explicit save in onboarding/settings | Edit versions; erase request |
| `session_snapshot` | Create shared snapshot | Dual independent confirm | Soft Signal / withdraw |
| `quiz_backup` | Optional own summary backup | Authenticated upsert | Delete row / erase request |
| `partner_quiz_share` | Share encrypted weather | Explicit share gate | Withdraw share |
| `partner_quiz_compare` | Joint view | Dual compare gates | Withdraw compare |
| `safety_report` | Process report | Submit action | Limited (investigation) |
| `privacy_notice` | Record notice ack | `accept_privacy_notice` | Superseded by new version |

### 4.3 Data subject rights (DSR)

| Right | Current engineering | Small-team DSR process |
| ----- | ------------------- | ---------------------- |
| Access / portability | In-app export + local inventory | User self-serve first; email fallback log |
| Rectification | In-app profile/touch edit | Same |
| Erasure | Device wipe + erasure **queue** | Ops checklist: confirm ID → export archive if required → wipe device guidance → fulfill server categories when legal allows |
| Restriction | Blocks / holds (safety) | Document when used for privacy vs safety |
| Objection | Soft Signal / decline request | Product UX already |
| Complaint | Not published | Placeholder until legal |

**SLA for small team:** Acknowledge DSR in **7 days**, complete or explain  
within **30 days** (GDPR-aligned target), track in a single `dsr-log.md` or  
Linear/GitHub project—not a heavy ticket system.

### 4.4 Safety features as privacy controls

| Safety feature | Privacy/PIMS value | Watch-outs |
| -------------- | ------------------ | ---------- |
| **Soft Signal** | Immediate stop; no reason required → reduces over-collection of sensitive narratives | Do not log “why” |
| **Consent Snapshot dual confirm** | Purpose-bound, session-specific processing | Snapshot retention policy |
| **RLS participant scope** | Confidentiality of shared session data | Service-role discipline |
| **Structured reports** | Minimized categories vs free-text dumps | Staff access control incomplete |
| **Blocks / matching holds** | Risk reduction | Purpose limitation: not “punishment data” for unrelated uses |
| **Age gate** | Children exclusion | Document legal age standard used |
| **Rate limits** | Abuse reduction | Avoid logging sensitive payloads |
| **Device-bound encryption** | Confidentiality of notes | Recovery/loss of access must be explained to users |

**Design principle:** Safety processing (reports, restrictions) should have  
**explicit purposes** in the RoPA and must not silently expand into marketing,  
ranking, or public “safety scores” (constitution).

## 6. Lightweight privacy risk register (start here)

Rate impact × likelihood simply (L/M/H). Treat **H** before external alpha.

| ID | Risk | Impact | Likelihood | Treatment (existing / planned) |
| -- | ---- | ------ | ---------- | ------------------------------ |
| R1 | Leak of Touch Language / private notes | H | L–M | Encryption, RLS, no logs, Face ID |
| R2 | Snapshot misused as permanent consent | H | M | Product copy + lifecycle invalidation |
| R3 | Partner quiz plaintext on server | H | L | E2E + relay refuse-list |
| R4 | Incomplete erasure after user request | H | H today | Queue + wipe; hard-delete blocked—communicate honestly |
| R5 | Staff over-access to reports | H | M | Staff auth + least privilege (open) |
| R6 | Processor breach (hosting) | H | L | DPA, vendor review, encryption in transit |
| R7 | Demo user enters real intimate data | M | M | Demo warnings; local wipe |
| R8 | Export overshares peer data | H | L | Export design excludes peer private notes |
| R9 | Retention forever by default | M–H | M | Retention schedule decision (blocked) |
| R10 | ND Mode misread as health data special category | M | L | Policy: preference only, not clinical |

Update this table when shipping features that process new PII categories.

## 7. Phased roadmap (small team)

Assume **1 founder + intermittent engineering agents**. Prefer **evidence in the  
repo** over parallel policy binders.

### Phase 0 — Foundation pack (1–2 weeks of calendar time, low hours)

**Goal:** Be able to answer “what do we process and why?” without inventing.

| # | Task | Owner | Evidence |
| - | ---- | ----- | -------- |
| 0.1 | Adopt this document as PIMS roadmap | Founder | This file |
| 0.2 | One-page **scope** (controller, systems, out-of-scope) | Founder | §2 signed in DECISIONS or privacy ADR |
| 0.3 | **RoPA lite**: table of processing activities | Founder + eng | New `docs/privacy/ROPA.md` (or section in GDPR.md) |
| 0.4 | **SoA lite**: list control themes Apply/Partial/N/A | Eng | Spreadsheet or `docs/privacy/SOA.md` |
| 0.5 | Subprocessor draft list | Founder | Appendix in Privacy Policy |
| 0.6 | Link Settings → Privacy surfaces (done) | Eng | App |

**Exit:** You can brief a privacy lawyer in 30 minutes with links only.

### Phase 1 — Rights & consent operations (2–4 weeks)

**Goal:** Operational DSR + consent evidence without full hard-delete.

| # | Task | Owner | Evidence |
| - | ---- | ----- | -------- |
| 1.1 | DSR runbook (export, wipe, erasure queue, SLAs) | Founder | `docs/privacy/DSR_RUNBOOK.md` |
| 1.2 | Erasure fulfillment checklist (what can be deleted now vs blocked) | Eng + founder | Runbook + KNOWN_LIMITATIONS |
| 1.3 | Privacy notice versioning discipline (bump `PRIVACY_NOTICE_VERSION` on material change) | Eng | `privacyContent.ts` + acceptances |
| 1.4 | Consent register published (§4.2 table maintained) | Eng | This file or ROPA |
| 1.5 | Export completeness review (touch + consent versions always present when exist) | Eng | Test + export schema notes |
| 1.6 | Soft Signal / withdraw copy audit (no pressure for reasons) | Product | UX review note |

**Exit:** A real user can export, wipe device, request erasure; ops can respond  
consistently.

### Phase 2 — Privacy by design hardening (ongoing with each milestone)

**Goal:** Controls that scale with features.

| # | Task | Owner | Evidence |
| - | ---- | ----- | -------- |
| 2.1 | PR template checkbox: classification + purpose + retention impact | Eng | PR template |
| 2.2 | CI/log hygiene checks for sensitive keys | Eng | Lint/test |
| 2.3 | Snapshot & session retention decision | Legal + founder | ADR |
| 2.4 | Staff moderation auth + access logging | Eng | ADR + migration |
| 2.5 | Processor DPAs (Supabase minimum) | Founder/legal | Signed DPA store |
| 2.6 | DPIA for “stranger matching + intimate preferences” before public beta | Privacy reviewer | DPIA doc |

**Exit:** No new sensitive field ships without classification + purpose.

### Phase 3 — Destructive rights & production readiness (external blockers)

**Goal:** Align with GDPR erasure and certification readiness.

| # | Task | Owner | Evidence |
| - | ---- | ----- | -------- |
| 3.1 | Named legal privacy counsel | Founder | Engagement |
| 3.2 | Approved hard-delete / anonymization design for `auth.users` cascade | Legal + eng | ADR + migration |
| 3.3 | Production retention jobs owned and monitored | Eng | Runbooks + alerts |
| 3.4 | Public controller identity, contact, complaint path | Legal | Privacy Policy |
| 3.5 | Incident response including privacy breach notification thresholds | Founder + eng | IR playbook |
| 3.6 | Optional: ISO 27001 base controls gap assessment | Consultant | Report |

**Exit:** Honest path to “we can fulfill erasure end-to-end.”

### Phase 4 — Certification path (optional, post product-market fit)

Only if commercially justified:

1. Engage ISO 27001/27701 consultant for gap assessment.  
2. Implement missing ISMS essentials (asset inventory, access reviews, supplier  
   management, internal audit, management review)—**scaled to team size**.  
3. External Stage 1 / Stage 2 audit.  
4. Maintain annual surveillance.

**Small-team warning:** Certification cost and process often exceed early-stage  
value. Prefer **Phase 0–2 evidence** until private alpha succeeds.

## 10. Minimum viable “PIMS week” (founder + one engineer)

If you only do **five things**:

1. **Name a privacy owner** (founder) in DECISIONS.md.  
2. **Freeze RoPA lite** (one table of processing activities).  
3. **DSR runbook** (export → wipe → erasure request → what we cannot yet delete).  
4. **Sign Supabase DPA** (or equivalent).  
5. **One privacy risk review** of touch/consent/snapshots/safety before expanding discovery.

These five create more certification-ready evidence than months of unfocused  
policy writing.

## 11. Anti-patterns (avoid)

- Claiming “ISO 27701 compliant” without audit/certification.  
- Building a parallel policy wiki disconnected from the repo (breaks durable-agent rule).  
- Treating session Soft Signal as a marketing opt-out only—or vice versa.  
- Expanding discovery fields without classification updates.  
- Silent hard-delete of safety evidence when legal hold may apply—use a  
  **documented** fulfill path.  
- Collecting “neurodivergent” as a health data field—keep ND Mode as a  
  **device preference**, not a clinical attribute.
